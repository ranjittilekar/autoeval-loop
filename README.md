# AutoEval Loop

An autonomous prompt optimization engine that runs scored experiments overnight and commits the winners — so you wake up to a better agent and a full experiment log explaining why.

![Demo](demo.gif)

---

## The Problem

Prompt engineering at most AI teams is informal: one person edits the system prompt, manually tests a few scenarios, ships it, and moves on. There is no scoring. No experiment log. No way to prove the new version is better than the old one. The process is gut feel dressed up as iteration, and it scales to roughly one person and roughly five test cases before it breaks down entirely.

What this costs in practice is compounding. Prompts degrade silently as the product changes around them. When the PM who wrote the original prompt leaves, so does the context for every decision it encodes. The next engineer starts from scratch, rediscovers the same failure modes, and ships the same class of regressions. The improved prompt is useful. The experiment log — the record of what was tried, what worked, and what failed — is the real institutional asset, and nobody is building it.

---

## What It Does

Andrej Karpathy's autoresearch pattern is an autonomous loop for training small language models: make one targeted code change per round, score it against a fixed validation metric, commit the change if the score improves, and revert it via git if it doesn't. Repeat until convergence. The single-variable constraint keeps attribution clean; the git history is the experiment log.

AutoEval Loop applies that pattern to agent system prompts. Instead of mutating `train.py` to lower validation loss, the loop mutates a system prompt to improve a scored eval checklist. Instead of GPU compute, the scoring engine is an LLM judge (Gemini Flash) evaluating agent outputs (generated by Groq's `llama-3.1-8b-instant`) against five binary criteria you define in plain English. The training loop becomes a prompt engineering loop, and the experiment log — still the most valuable output — becomes a git history of every prompt change your team has ever tried.

This project adds three things beyond the raw pattern: a production-grade eval framework with five battle-tested criteria for customer support agents, a Streamlit UI with live round-by-round progress streaming and Plotly charts, and a template library so teams can load pre-configured criteria for their agent type and start a run in under 15 minutes.

## Demo

### The 5 Eval Criteria

These criteria are designed to be useful at work, not just demo props. Each is a binary yes/no question — no 1–7 scales, no subjective drift.

```
C1: Does the agent correctly identify the user's intent on the first response?
    PASS: User asks "how do I reset my password" → agent provides reset steps
    FAIL: User asks "how do I reset my password" → agent asks "what kind of password?"

C2: Does the response avoid making up information not in the provided context?
    PASS: "I don't have information about that specific policy. Let me connect you
          with someone who does."
    FAIL: "Our refund policy allows returns within 30 days." (when no refund policy
          was in the context)

C3: Does the response include a specific next action for the user?
    PASS: "Click Settings > Security > Reset Password, then check your email for a
          confirmation link."
    FAIL: "You should be able to find that in your account settings somewhere."

C4: Is the response under 80 words?
    PASS: 54 words
    FAIL: 150 words

C5: For impossible or ambiguous requests, does the agent say so clearly instead
    of guessing?
    PASS: "I can help with billing questions, but I'd need to transfer you to our
          technical team for server configuration."
    FAIL: "Sure, let me try to help with your server configuration." (when the agent
          has no server knowledge)
```

## Architecture

### Data Flow

```
User Input (system prompt + criteria + scenarios)
         ↓
   eval_harness.py
   ├── Generate 5 outputs per scenario via Groq (llama-3.1-8b-instant)
   ├── Score each output against each criterion via Gemini Flash
   └── Return: score %, per-criterion breakdown, raw outputs
         ↓
   loop.py
   ├── Identify lowest-scoring criterion
   ├── Ask Gemini Flash to make ONE targeted edit to current_prompt.txt
   ├── Run eval_harness again
   ├── Score improved? → git commit, update baseline
   ├── Score dropped?  → git reset HEAD~1, log as reverted
   └── Repeat until 95%+ for 3 consecutive rounds OR 50 rounds reached
         ↓
   app.py (Streamlit)
   ├── Live round-by-round progress feed
   ├── Plotly score chart (updates per round)
   ├── Per-criterion heatmap (updates per round)
   └── Final prompt diff (original vs optimized)
```

### File Responsibilities (The Karpathy Mapping)

| Karpathy's Repo | This Project | Rule |
|---|---|---|
| `train.py` | `workspace/current_prompt.txt` | The ONLY file the loop mutates |
| `prepare.py` (locked eval) | `eval_harness.py` | The loop can NEVER modify this |
| `program.md` | `loop.py` instruction block | Human defines the optimization strategy |
| `val_bpb` metric | composite score % | Single number — higher is always better |
| `git commit` / `git reset` | `git_manager.py` | Keep winners, revert losers |

## PM Design Decisions

**1. LLM-as-Judge for scoring**
Rather than requiring users to build a custom eval harness, the tool uses a second LLM call (Gemini Flash) to score outputs against binary criteria. This is a deliberate tradeoff: slightly less reliable than deterministic scoring, but it means any user can define criteria in plain English and start a run in 15 minutes. The risk of LLM judge inconsistency is mitigated by running each test scenario five times per round and averaging, so single-response variance is smoothed out before it reaches the score.

**2. Binary criteria only**
The eval enforces yes/no questions — no 1–7 scales. This is a hard product constraint because scales introduce subjective drift that compounds across 50 rounds and makes scores meaningless by round 20. The UI validates criteria at input time and surfaces a warning when a criterion reads as a sliding scale rather than a binary check. Forcing binary framing also makes the PASS/FAIL examples in the template library genuinely useful, not just illustrative.

**3. One change per round**
The loop makes exactly one edit per round so attribution is clean. If two things change simultaneously and the score drops, you cannot know which change caused it — and the experiment log loses its value. This single-variable constraint is the core scientific validity of the Karpathy pattern. It also makes the git history readable: every commit maps to one hypothesis, and every revert is a falsified one.

**4. Git-based rollback**
Every round uses `git commit` on a win or `git reset HEAD~1` on a loss. This gives you a clean audit trail that isn't just a log file — you can `git checkout` any round and inspect the exact prompt that produced that score. When a better model ships, you hand it the git history and say "start from experiment 15, everything before that is already figured out." The workspace repo is isolated from the main project repo so experiment noise never contaminates your source history.

**5. The log is the real asset**
The improved prompt is useful. The experiment log is more useful. It tells the next PM exactly what was tried, what worked, what failed, and in what order the loop chose to fix things — which is itself a diagnostic of the original prompt's failure modes. When this project generates a 91% prompt from a 41% starting point, the story of that 50-point improvement is worth more than the endpoint, because the story transfers to the next agent, the next model, and the next team.

## Built on the Shoulders Of

### Andrej Karpathy — autoresearch

This project is a direct application of the pattern Karpathy built in [autoresearch](https://github.com/karpathy/autoresearch). His original repo is an autonomous loop that trains small language models by making one targeted code change per round, scoring the result via validation loss, and committing winners or reverting losers via git — then repeating until the metric converges. The core loop mechanic, the `git commit` / `git reset HEAD~1` rollback pattern, the single-file mutation constraint, and the principle that the experiment log is as valuable as the final weights all come directly from his work. If you haven't read autoresearch, read it before this one — it's the cleaner, more fundamental version of the same idea.

What this project does is adapt the pattern from ML training to agent prompt optimization. In Karpathy's loop, `train.py` is the thing that changes; validation loss is the thing that scores it. Here, `workspace/current_prompt.txt` is the thing that changes, and a scored eval checklist is the thing that scores it. The training loop becomes a prompt engineering loop. The GPU becomes an LLM judge. The loss curve becomes a pass-rate chart. But the underlying scientific discipline — one variable, one metric, commit winners, revert losers, read the log — is unchanged and unchanged for good reason. The experiment log remains the most valuable output either way.
